## The goal (in Mike's words)

Move from "development canon as prose rules" → "development canon as runnable generators". Want to re-run a generator across any mist83/* repo and have all boilerplate regenerate while user code stays untouched (library-driven, scaffolding refusable). End state is `dag.[base-url]` visual composer where nodes are components and edges are wiring.

**Why:** Mike currently tells agents "follow the canon" and they guess wrong — he ends up pointing to an example site anyway. Generators would make the canon executable instead of suggestive.

**How to apply:** When Mike picks this up again, do NOT start writing generators. Nail the component manifest contract FIRST. Order matters: contract → CLI → one component retrofit → library-paired generators → DAG composer wiring.

## Empirical drift snapshot (from 2026-04-17 analysis of ~/Code)

- 196 repos total under mist83/*
- 179 have `mullmania.site.json`
- 161 have `deploy.sh` — 157 are byte-identical (remote-sourced pattern works ✅)
- 141 copy-paste `docs/index.html`, 138 copy-paste `docs/frontdoor.json` (these drift)
- 167 have `.github/workflows/` (likely also drift)
- Only 4 have `AGENTS.md` — inconsistent
- Only 12 have `sitemap.json` — emerging pattern

**Lesson:** anything sourced from one URL (deploy.sh) has zero drift. Anything copy-pasted as files drifts immediately.

## The three things that turn rules into components

1. **Component manifest** — each fragment (ui-shell, favicon, frontdoor, github-deploy, bug-beacon, f3, etc.) becomes a first-class unit with `name`, `version`, typed `inputs`, `outputs` (file list), `apply()`, `check()` (drift), `remove()`.

2. **Per-repo lockfile** — extend existing `mullmania.site.json` (don't add a second file) with `"components": { "ui-shell": "3", "github-deploy": "2" }`. Already has `canon.version`, so `components.*` is the natural sibling.

3. **Marked regenerable regions** — every generated file gets `<!-- @generated by mullmania-scaffold:ui-shell@3 — safe to regenerate -->`. User code lives outside markers. This is the "library-driven, scaffolding refusable" property.

## Build order (when Mike picks this up)

1. Draft the component manifest schema (+ idempotence contract written down, enforced by shared test harness)
2. Build `scaffold` CLI: `add`, `upgrade`, `check`, `remove` (likely a new `mist83/scaffold` repo — the `development-canon` repo is retired/archived)
3. Component #0: `mullmania-site` — de-drifts 138 frontdoor.json copies
4. Components #1-3: `ui-shell`, `github-deploy-workflow`, `favicon-set`
5. Library-paired generators: `bug-beacon`, `f3`, `signal-argh` each ship their own. `npx bug-beacon init .` adds FAB + lockfile entry. Library owns its own scaffolding; canon prose for these disappears.
6. Retrofit loop: point the existing audit harness (now the `audit-an-app` kata on katai, `katai.mullmania.com`; the `audit-app.mjs` script formerly lived in the archived development-canon repo) at `scaffold check --json` across all repos → health dashboard.
7. DAG composer: add "read components, emit edits to mullmania.site.json" endpoint to `dag/backend`. Visual editor is a lockfile editor in disguise.

## What Mike is underweighting (flag if he forgets)

- **Idempotence contract must be written down and test-harness-enforced.** Every generator: safe to rerun, identical output from identical inputs, never touches bytes outside markers. Without formal contract, generators drift like templates.
- **Uninstall semantics** — `scaffold remove bug-beacon`. Most scaffold systems only add. He'll want removal.
- **Component registry at a URL** — `https://scaffold.[base-url]/registry.json`. Same pattern as existing `canon.json`.
- **Per-component proof** — component publishes the diff it would produce against a fresh repo. Then `check` is just "re-run and diff against published proof."
- **AGENTS.md only in 4/196 repos** — should itself become a generated+pinned component.

## Prior art (closest → furthest)

- **Backstage scaffolder** (Spotify) — closest to DAG + catalog + component end-state; heavy/YAML/org-scale
- **Nx generators + executors** — best shape (generators write files, executors run tasks, JSON schemas for inputs); monorepo-centric while Mike has 196 separate repos
- **Angular Schematics** — best precedent for "replayable, safe-to-rerun, upgrade command"
- **Terraform modules / CDK constructs** — the versioned typed-input composable model
- **Cookiecutter / create-*-app** — one-shot, no upgrade path
- **Dagger.io** — graph pipelines, closest to DAG vibe but build/CI-focused
- **Yeoman** — historical, dead, bad UX

**The gap nobody has filled:** library-ships-own-generator + DAG-composer + personal-studio-scale (not company). That's Mike's actual white space.

## Honest frame for Mike

He has the **vocabulary** (canon, site.json, ui framework, deploy.sh-as-URL, dag repo). He's missing the **verbs** (`add`, `upgrade`, `check`, `remove`) as real code. Three focused sessions of Nx-schematic-shaped CLI + one component retrofit bridges the whole distance.
