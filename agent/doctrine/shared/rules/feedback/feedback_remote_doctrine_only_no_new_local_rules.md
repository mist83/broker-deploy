The cloud doctrine is the brain. Local rule piles are migration debt.

## Rule

- Do not author new standing policy in local `AGENTS.md`, `CLAUDE.md`, `MEMORY.md`, skill files, command files, hook bodies, local plugin registries, or local automation registries.
- Local instruction-shaped files may exist only as bootstrap pointers, safety fuses, generated projections, or caches.
- If you discover other local rule surfaces that look like hidden doctrine, say so plainly. Name the path, explain why it is a drift risk, and recommend migration or deletion.
- Do not silently tolerate or extend a stray local rule pile just because it already exists.
- If behavior needs to change, change the remote doctrine at `agent.mullmania.com` instead of inventing a local workaround.

## Why

The operator wants one cloud-managed brain, not a fresh local rules ecosystem every time a runtime grows a new surface.

If there is more local source-of-truth than the tiny foothold the runtime absolutely requires, that is not "helpful flexibility." It is drift.
