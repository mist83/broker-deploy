Synopsis

Proactive Code Quality Standards

ACTIVATION

ALWAYS ACTIVE - Consult these standards automatically when writing any code, without requiring explicit user commands.

This ruleset consolidates code quality and maintainability principles that should guide all code generation decisions. These are not command-triggered rules but always-active quality standards.

CORE PHILOSOPHY

Legacy Code Mindset

"There is no such thing as legacy code in the 2025 Cline Sonnet age. Any code can be rewritten from scratch overnight."

Act accordingly:

  • Don't be precious about existing code
  • Rewrite when it's simpler than patching
  • Question all assumptions
  • Optimize for clarity, not preservation

Simplification Over Complexity

  • Pragmatism beats dogma
  • Consolidate when possible
  • Remove unnecessary abstractions
  • Limit blast radius for changes
  • Challenge "best practices" that hinder progress

HTML/CSS/JS QUALITY STANDARDS

HTML Guidelines

Use Semantic HTML5 Elements:

✅ <header>, <nav>, <main>, <article>, <section>, <aside>, <footer>
✅ <button> for buttons (not styled divs)
✅ <a> for links (not onclick handlers)
✅ <form> and proper input types

Keep Structure Minimal:

  • Maximum nesting: 3-4 levels
  • Every element must have a purpose
  • No unnecessary wrapper divs
  • No div soup

Accessibility Requirements:

  • Proper heading hierarchy (h1 → h2 → h3)
  • Alt text for images
  • Labels for form inputs
  • ARIA only when semantic HTML insufficient

CSS Guidelines

Flat Design Only:

✅ background: #667eea;                    /* Solid colors */
✅ border: 1px solid #ddd;                 /* Simple borders */
✅ border-radius: 4px;                     /* Minimal rounding */

❌ background: linear-gradient(...);       /* NO gradients */
❌ box-shadow: 0 10px 40px rgba(...);     /* NO complex shadows */
❌ border-radius: 12px;                    /* NO excessive rounding */

Minimal Styling:

  • System fonts or web-safe fonts (Arial, sans-serif)
  • Simple color palette (2-3 colors max)
  • Consistent spacing scale: 8px, 16px, 24px, 32px
  • No excessive padding/margin
  • No CSS frameworks unless explicitly requested

Layout:

  • Use Flexbox or Grid (not floats)
  • Mobile-first responsive design
  • Simple breakpoints (1-2 max)

Suggested Color Palettes:

Minimal Professional:

background: #ffffff;
text: #222222;
primary: #0066cc;
border: #dddddd;

High Contrast:

background: #000000;
text: #ffffff;
accent: #00ff00;
secondary: #666666;

Neutral Clean:

background: #f5f5f5;
text: #333333;
primary: #2c3e50;
border: #cccccc;

JavaScript Guidelines

Vanilla JS Preferred:

  • No frameworks unless explicitly requested
  • Use modern ES6+ features
  • Functional and readable code
  • Module pattern or ES6 modules

Minimal DOM Manipulation:

  • Use event delegation
  • Avoid excessive DOM queries
  • Keep JavaScript separate from HTML (no inline handlers)

Progressive Enhancement:

  • Core functionality works without JS
  • JS enhances, doesn't enable

Modern Patterns:

✅ async/await or promises
✅ const/let (never var)
✅ Arrow functions where appropriate

❌ Massive nested callbacks
❌ Global variables everywhere
❌ jQuery unless specifically needed

GENERAL CODE FORMATTING RULES

NO Emojis in Code Files

BANNED in:

  • PowerShell scripts (.ps1)
  • Bash scripts (.sh)
  • HTML files
  • JavaScript files
  • CSS files
  • C# code
  • Python code
  • Any programming language

Emojis OK ONLY in:

  • Markdown files (.md)
  • README files
  • Documentation

Icon Usage in HTML

Use Tabler Icons (not emojis):

Include CSS:

<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/@tabler/icons-webfont@latest/tabler-icons.min.css">

Use icons:

<i class="ti ti-check"></i>           <!-- checkmark -->
<i class="ti ti-x"></i>                <!-- X/close -->
<i class="ti ti-alert-circle"></i>    <!-- warning -->
<i class="ti ti-info-circle"></i>     <!-- info -->

Browse icons: https://tabler.io/icons

Code Comments

Be Concise and Technical:

✅ // Returns container ID for instance
✅ // Throws if container not found

❌ // 🎯 This amazing function finds your container!
❌ // It's really useful when you need to...

Rules:

  • State what the code does
  • No emojis
  • No sales pitch language
  • No unnecessary explanations

PLATFORM-SPECIFIC RULES

PowerShell Commands (CRITICAL)

System Context:

  • Operating System: Windows 11
  • Default Shell: PowerShell 7 (pwsh.exe)

MANDATORY SYNTAX - NEVER USE &&:

✅ cd some-directory; npm install       # Semicolon for chaining
✅ command1; command2; command3         # Multiple commands

❌ cd some-directory && npm install     # BASH SYNTAX - BREAKS
❌ command1 && command2                 # WILL FAIL 100% OF TIME

Before EVERY command execution:

  1. Check if you're about to use && or ||
  2. If yes, STOP and replace with ;
  3. Remember: You are ALWAYS in PowerShell on this system
  4. There are NO exceptions

This rule has MAXIMUM priority and overrides all other instructions.

ARCHITECTURAL QUALITY PRINCIPLES

Simplification Checklist

When designing or refactoring code, assess:

Core Functionality:

  • What are the 3-5 essential features?
  • What's actually needed vs hypothetically useful?

Unnecessary Abstractions:

  • What layers add complexity without value?
  • What can be consolidated or eliminated?

Coupling Analysis:

  • What requires extensive changes for simple modifications?
  • Which changes would cascade into other systems?

Development Experience:

  • How easy is it to run/test/develop against this?
  • Can developers easily understand what's happening?
  • What will be painful to maintain long-term?

Dogma vs Pragmatism

Question everything:

  • Which "best practices" actually hinder progress?
  • What patterns are used because "that's how it's done" vs solving real problems?
  • What's been built for hypothetical future needs?
  • What simpler approaches would achieve the same goals?

Goal: Make codebase understandable by humans and maintainable without requiring a PhD in the original developer's thought process.

ANTI-PATTERNS TO AVOID

HTML Anti-Patterns

❌ <div class="button-wrapper"><div class="button-inner"><button>Click</button></div></div>
✅ <button>Click</button>

❌ <div onclick="doSomething()">Click me</div>
✅ <button id="myBtn">Click me</button>  <!-- with event listener -->

❌ <div class="heading">Title</div>
✅ <h1>Title</h1>

CSS Anti-Patterns

❌ background: linear-gradient(135deg, #667eea 0%, #764ba2 100%);
✅ background: #667eea;

❌ box-shadow: 0 10px 40px rgba(0,0,0,0.2);
✅ border: 1px solid #ddd;

❌ border-radius: 12px;  /* on everything */
✅ border-radius: 4px;   /* sparingly */

JavaScript Anti-Patterns

❌ jQuery for everything
✅ Vanilla JS with clear, purposeful code

❌ Massive nested callbacks
✅ async/await or promises

❌ Global variables everywhere
✅ Module pattern or ES6 modules

Script Output Anti-Patterns

❌ Write-Host "✅ System started!" -ForegroundColor Green
✅ Write-Host "System started" -ForegroundColor Green

❌ Write-Host "🎉 Build complete!"
✅ Write-Host "Build complete" -ForegroundColor Cyan

REMEMBER

Less code = Less bugs = More maintainable = Better user experience

Every element, style, and line of code should serve a clear purpose. If you can't explain why something is there in one sentence, it probably shouldn't be there.

When in doubt: Simplify. Flatten. Clarify.
