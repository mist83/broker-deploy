Shared UI Library

Canonical component and theme system hosted at ui.mullmania.com. Start from the runtime and shared helpers, but build with hierarchy, restraint, and clear section purpose instead of generic component piles.

Sitemap First

  1. Declare tabs, panels, and nested navigation in sitemap.json.
  2. Prefer preset, component, or componentSource over htmlSource.
  3. Drop into raw HTML only when the shared runtime genuinely cannot express the surface yet.

Consumer Design Rules

  1. Start with composition, not components.
  2. Give each section one job and one takeaway.
  3. Make the brand or product name the loudest text on branded pages.
  4. Default to cardless layout. Use cards only when the card itself is the interaction.
  5. Use no more than two typefaces and one accent color by default.
  6. Use utility copy for product UI and product copy for branded pages.

One-Line Start

<script src="https://ui.mullmania.com/ui.js"></script>

The runtime injects the current published default theme, Tabler icons, and the shared JS primitives automatically.

Published Default

  • Bare ui.js, data-ui-theme="active", and /active/* follow the published global default.
  • Use a concrete theme id like ocean or walmart when a site should stay pinned.
  • UI.setTheme() previews locally and does not publish a new default.

Branded Page Rules

  • Treat the first viewport like a poster.
  • Use one dominant visual idea in the hero.
  • Put the order in this sequence: brand, headline, support copy, CTA.
  • Avoid hero cards, stat strips, and floating dashboard clutter by default.

Product UI Rules

  • Default to calm surface hierarchy.
  • Use headings that orient the operator immediately.
  • Use spacing and typography before adding more chrome.
  • If a panel can become plain layout without losing meaning, remove the card treatment.

Typography Rules

  • One sans family for controls, forms, tables, and body text.
  • One display family only for high-impact moments.
  • Mono for labels, tabs, token names, and diagnostics.
  • Use scale and spacing before decorative effects.
  • Prefer semantic role classes or UI.text({ role: ... }) over page-specific selectors.

JS-First Rendering

<div id="app"></div>
<script src="https://ui.mullmania.com/ui.js"></script>
<script>
UI.ready().then(() => {
  UI.mount('#app', {
    title: 'Agent Workspace',
    subtitle: 'Operational UI with clear hierarchy.',
    sections: [
      UI.section({
        title: 'Overview',
        description: 'Use structure and spacing before adding more containers.',
        children: [
          UI.grid({
            columns: 2,
            children: [
              UI.stat({ label: 'Status', value: 'Live' }),
              UI.alert({
                tone: 'info',
                title: 'Working Rule',
                message: 'Do not default to a generic card mosaic.'
              })
            ]
          })
        ]
      })
    ]
  });
});
</script>

Primary Helpers

  • UI.mount(target, definition)
  • UI.text({ role: 'brand' | 'display' | 'ui-title' | 'body' | 'label' })
  • UI.section(options)
  • UI.stat(options)
  • UI.grid(options)
  • UI.card(options) when a card is truly the right interaction
  • UI.table(options)

Reject These Failures

  1. Generic SaaS card grid as the first impression.
  2. Weak brand presence in the first screen.
  3. Too many accent colors.
  4. Theme-specific hacks in generic work.
  5. Ad hoc local CSS for patterns that should live in the library.
  6. Using htmlSource when a sitemap-driven preset or component would do the job.
